## 📊 Detailed Marking Rubric

### Stage 1 — Proposal Defense (10 marks)

| Criteria | Excellent (9–10) | Good (7–8) | Satisfactory (5–6) | Needs Work (<5) |
|---|---|---|---|---|
| **Proposal Quality** | Clear problem, well-scoped, realistic goals | Good problem definition, minor scope issues | Vague objectives or over-ambitious scope | Unclear problem; scope undefined |
| **Project Plan** | Detailed timeline, milestones, roles assigned | Good timeline, some gaps | Partial plan; timeline missing detail | No realistic plan |
| **Budget/Resource Plan** | Full resource list, cost-conscious, feasible | Most resources identified | Partial resource identification | No resource planning |

---

### Stage 2A — Work Done (50 marks)

| Criteria | Excellent (9–10) | Good (7–8) | Satisfactory (5–6) | Needs Work (<5) |
|---|---|---|---|---|
| **System Design** | Complete ER, DFD, flowcharts, UI mockups aligned | Most diagrams present and consistent | Key diagrams present but incomplete | Missing or inconsistent design artifacts |
| **Thoroughness** | All planned features implemented and tested | Most features implemented | Core features only; major gaps | Incomplete implementation |
| **Understanding of Methods** | Demonstrates deep understanding in viva and report | Good understanding with minor gaps | Basic understanding | Unable to explain design choices |
| **Amount of Work** | Full-scale application with substantial code | Moderate scope, well-executed | Limited but functional | Very little usable code |
| **Level of Achievement** | Exceeds proposal goals | Meets all goals | Meets most goals | Falls significantly short |
| **Teamwork / Independence** | Clear evidence of contribution and coordination | Good contribution with minor coordination gaps | Acceptable contribution | Unequal or unclear contributions |
| **Problem Identification** | Proactively identifies and resolves issues | Identifies issues when prompted | Identifies obvious issues only | Unaware of project problems |
| **Planning Skills** | Sticks to milestone schedule; adapts plan proactively | Minor schedule drift; recovered | Moderate delays; partially recovered | Significant deviations from plan |

---

### Stage 2B — Documentation (20 marks)

| Criteria | Excellent (9–10 per 5) | Good | Satisfactory | Needs Work |
|---|---|---|---|---|
| **Report Organization** | Logical flow; standard format followed precisely | Mostly organized; minor gaps | Some sections out of order | Disorganized, hard to follow |
| **Writing Style** | Clear, professional, consistent academic tone | Mostly clear with minor errors | Understandable but informal | Poor grammar, unclear writing |
| **Completeness** | All required sections present and fully developed | Most sections complete | Key sections thin or missing | Major sections absent |
| **Readability** | Well-formatted, consistent fonts, headers, spacing | Mostly readable; minor formatting issues | Readable but inconsistent | Hard to read; poor layout |
| **Data/Results Analysis** | Clear presentation of outputs, screenshots, test cases, results discussed | Good analysis with minor gaps | Basic output screenshots | No results or analysis |

---

### Stage 3 — Viva Voce (20 marks)

| Criteria | Excellent (17–20) | Good (13–16) | Satisfactory (9–12) | Needs Work (<9) |
|---|---|---|---|---|
| **Technical Understanding** | Explains every design and code decision clearly | Good explanation with minor gaps | Explains main flow; struggles with details | Cannot explain own code/design |
| **Demonstration** | Live demo runs flawlessly; all features shown | Demo mostly smooth; minor glitches | Demo partial; some features not working | Demo fails or not ready |
| **Response to Questions** | Answers confidently and accurately | Mostly accurate; minor hesitation | Some correct answers; needs prompting | Cannot answer evaluator questions |
| **Problem Analysis** | Articulates limitations and future scope thoughtfully | Identifies limitations | States one or two limitations | Unaware of system limitations |

## 💡 Project Selection Guidelines

### Suitable Project Domains

| Domain | Example Projects |
|---|---|
| Education | Student Result Management, Library Management, Online Quiz System |
| Healthcare | Patient Record System, Pharmacy Management, Appointment Scheduling |
| Business | Inventory Management, Billing System, HR Payroll |
| Community | Hostel Management, Cooperative Records, Voter Management |
| E-commerce | Online Shopping Cart, Product Catalogue with Cart |
| Agriculture | Crop Record System, Fertilizer Stock Management |
| Government Services | Municipality Record, Land Record System |

### Technology Stack Options

| Layer | Options |
|---|---|
| **Frontend (Desktop)** | Java Swing, C# WinForms, Python Tkinter/PyQt |
| **Frontend (Web)** | HTML + CSS + JavaScript, React, Bootstrap |
| **Backend** | PHP, Python (Flask/Django), Java (Spring), Node.js |
| **Database** | MySQL, MS SQL Server, SQLite, PostgreSQL |
| **IDE / Tools** | NetBeans, IntelliJ, VS Code, Visual Studio, XAMPP |
| **Version Control** | Git + GitHub |
| **Documentation** | MS Word, LaTeX |
| **Diagrams** | draw.io, Lucidchart, MySQL Workbench |

### What Makes a Good Project?

- Solves a **real problem** for an identifiable group of users
- Has **clear scope** — completable in one semester by 1–2 people
- Uses concepts from at least **3 courses** studied (DBMS, OOP, Networking basics, etc.)
- Has a **proper database** (not a file-based system)
- Includes **user authentication**, at least two user roles, and meaningful data operations
- Goes **beyond the syllabus** where appropriate (not just copying lab exercises)

## ⚠️ Common Mistakes to Avoid

| Mistake | Better Approach |
|---|---|
| Choosing too large a scope | Start with 3–4 core modules; add features only after core works |
| Starting coding before design | Always complete ER diagram and DFD before writing any code |
| No version control | Commit to GitHub from Day 1; supervisor may ask to see commit history |
| Writing report at the end | Update report incrementally every week from Week 7 onward |
| Demo that crashes in viva | Test on the exam machine beforehand; have backup screenshots |
| Copied code with no understanding | Be able to explain every line; evaluators will ask about specific code |
| Missing normalization in DB | Always normalize to 3NF; justify any denormalization |
| No user manual | Include a brief user guide as an appendix |

## 🔑 Tips for Viva Success

1. **Know your own code** — Be prepared to explain any function, class, or query you wrote.
2. **Run the demo yourself** — Do not ask someone else to operate the system during your presentation.
3. **Know your ER diagram cold** — Evaluators frequently ask "Why did you model it this way?"
4. **Prepare for failure scenarios** — Show what happens when invalid data is entered; demonstrate validation.
5. **Know your limitations** — Saying "this feature is not implemented because..." shows maturity.
6. **Prepare future scope** — Have 3–4 genuine enhancement ideas ready.
7. **Practice your intro** — Spend 2 minutes explaining what the system does, for whom, and why it matters.
