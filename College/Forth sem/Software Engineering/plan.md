## 📌 Course Description

This course provides the **fundamental concepts and standards of software engineering** so that students will be able to:
- Develop software using global standards
- Handle software projects professionally
- Identify and document system requirements
- Design, test, and maintain software systems

> It presumes a general understanding of computers and programming covered in the first and second semester of the degree.

## 📐 Key Concepts Quick Reference

### Software Process Models — Comparison

| Model | Type | Best For | Key Feature |
|---|---|---|---|
| **Waterfall** | Sequential | Well-defined requirements | Phase-by-phase, no overlap |
| **Evolutionary** | Iterative | Unclear requirements | Build, feedback, improve |
| **Incremental** | Iterative | Large systems | Deliver in increments |
| **Spiral** | Risk-driven | High-risk projects | Risk analysis at each loop |
| **Agile / XP** | Iterative | Changing requirements | Short sprints, collaboration |
| **RAD** | Fast | Business systems | 60–90 day cycle |
| **RUP** | Iterative | Enterprise projects | 4 phases, 6 disciplines |
| **CBSE** | Component-based | Reuse-focused | Assemble from components |

---

### Types of Software Requirements

| Type | Description | Example |
|---|---|---|
| **Functional** | What the system should do | "User can log in with email/password" |
| **Non-Functional** | How the system should perform | "System should respond within 2 seconds" |
| **Domain** | Requirements from application domain | "Medical records must follow HIPAA standards" |
| **User Requirements** | High-level, user-readable statements | "The system shall allow users to place orders" |
| **System Requirements** | Detailed, technical specifications | "System stores order data in a relational DB" |

---

### Cohesion vs Coupling

| Concept | Goal | Best Type | Worst Type |
|---|---|---|---|
| **Cohesion** | High cohesion preferred | Functional Cohesion | Coincidental Cohesion |
| **Coupling** | Low coupling preferred | Data Coupling | Content Coupling |

---

### CMM Levels

| Level | Name | Description |
|---|---|---|
| **Level 1** | Initial | Unpredictable, poorly controlled |
| **Level 2** | Repeatable | Basic project management in place |
| **Level 3** | Defined | Documented and standardized processes |
| **Level 4** | Managed | Processes measured and controlled |
| **Level 5** | Optimizing | Continuous process improvement |

---

### Types of Software Maintenance

| Type | Description | Example |
|---|---|---|
| **Corrective** | Fix bugs after delivery | Patching a login crash |
| **Adaptive** | Adapt to new environment | Update for new OS version |
| **Perfective** | Add/improve features | Faster search algorithm |
| **Preventive** | Prevent future failures | Code refactoring |

---

### Software Cost Estimation — COCOMO Model

| Mode | Description | Team Size |
|---|---|---|
| **Organic** | Simple, small software | Small, experienced teams |
| **Semi-detached** | Medium complexity | Mixed-experience teams |
| **Embedded** | Complex, strict constraints | Large, strict teams |
